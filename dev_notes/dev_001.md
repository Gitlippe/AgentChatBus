# Agent 进度可见性 —— 调研结论

> 记录于 2026-03-04。经过完整的技术调研，本功能已确认搁置。

---

## 需求

让 AgentChatBus 的 Web UI 能实时看到 Agent 正在做什么（读文件、编辑代码、思考等），
而不是只有"在线/离线"两种状态。

---

## 调研结论：当前架构不可行 ❌

### 根本原因

Cursor / Claude Desktop 等商业 IDE 中的 LLM Agent，
其"工作中"的内部活动对 AgentChatBus 来说**架构上永久不可见**。

```
[msg_wait 返回] ──────── 黑暗区 ──────── [msg_post 发出]
                  LLM 读文件、写代码、思考
                  全部使用 IDE 原生工具
                  完全不经过 AgentChatBus
```

### 逐一排除的方案

| 方案 | 排除原因 |
|------|---------|
| `notifications/progress` 标准协议 | 必须绑定 active request 的 progressToken，黑暗区内无 active request，协议层禁止 |
| SSE 连接存活检测 | Agent 结束后 SSE 连接仍保持数分钟，完全不可靠 |
| `msg_wait` 返回时间戳推断 | Agent 若崩溃/卡死/用户关闭 Cursor，计时可达数小时，完全失真 |
| 新增 `agent_update_status` Tool | 要求 LLM 主动调用，增加认知负担；且无法覆盖 IDE 原生工具执行阶段 |
| 自定义脚本 Agent 上报 | 脚本无 LLM 智能，对本工程无实际价值 |
| `msg_post` 附带 `activity` 字段 | 事后回溯，活动结束后才上报，无实时价值 |

### 协议边界说明

这不是工程实现能力的问题，而是 **MCP 协议的观测边界**：

- IDE 原生工具（`view_file`、`write_to_file`、`run_command` 等）的执行，
  走 IDE 内部沙盒，完全独立于 AgentChatBus 的 MCP 通道。
- 除非 LLM 主动调用我们的 Tool 汇报（有认知负担），
  或使用我们控制的自定义客户端（失去 LLM 智能），否则该信息永久不可达。

---

## 当前可保持的能力

**`msg_wait` 的"等待中"状态仍然准确可用**：
- Agent 在调用 `msg_wait` 挂起期间，Bus 可准确知道它处于等待状态。
- 这是唯一有意义的、可靠的、零负担的状态可见性。

---

## 决策

> **搁置。不实现。**
>
> 待将来出现以下条件之一时，可重新评估：
> - 商业 IDE 客户端（Cursor 等）原生暴露工具执行事件钩子；
> - 工程转向使用我们控制的自定义 LLM 客户端运行时。
