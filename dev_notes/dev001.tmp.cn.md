# 当前实现说明（A 检测 + B 决策）

本文档说明当前系统中“管理员协调”机制的真实行为，重点是：

- A：检测触发（系统自动）
- B：人类决策（仅在需要时）

当前版本已经调整为三条路径：

1. `单在线且该 agent 就是当前管理员`：发接管确认卡（human-only）。
2. `多在线且全员 wait 超时`：不发确认卡，发 human-only 超时说明 + 发管理员协调指令。
3. `无法通知可行动管理员`：只发 human-only 风险告警（可能全员掉线）。

---

## 1. 一句话先讲清楚

当前机制不是“谁掉线就自动换管理员”。

当前机制是：

1. 当线程中“当前在线参与者”全部进入 `msg_wait` 并持续达到 `timeout_seconds`，系统触发一次协调处理。
2. 根据在线形态选择不同动作：确认卡、通知管理员、或仅告警人类。
3. 只有显式人类决策（`switch/keep/takeover/cancel`）才会执行对应后续动作。

---

## 2. 角色与职责

### A（检测层）做什么

1. 追踪每个线程的 wait 状态。
2. 判断是否满足“全员 wait 超时”。
3. 识别当前管理员是否在线且可通知。
4. 发确认卡或通知消息（按场景分流）。
5. 不直接做“隐式切换”。

### B（决策层）做什么

1. 接收人类在卡片上的选择。
2. 校验 `source_message_id` 和 `ui_type` 是否匹配。
3. 执行 `switch/keep/takeover/cancel`。
4. 回写决策结果并把卡片标记 resolved。

---

## 3. timeout 到底控制什么

`timeout_seconds` 控制的是“协调检测窗口”，不是“最后一条消息时间”。

判断方式：

1. 系统取“当前在线参与者集合”。
2. 只有当这个集合里的所有人都在 `msg_wait`，才进入计时。
3. 计时锚点是这个集合中“最后进入 wait 的那个人”的 `entered_at`。
4. 从锚点开始经过 `timeout_seconds`，触发协调处理。

如果中途有人退出 wait，条件被打破，后续需要重新满足“全员 wait”再计时。

---

## 4. human-only 的含义

`human_only` 消息用于治理，不用于驱动 agent 自动行为：

1. 前端会明确提示 human-only。
2. 工具层会过滤 human-only，agent 读不到这类消息。
3. 发给管理员执行工作的消息不会标 human-only，才能唤醒其 `msg_wait`。

---

## 5. 关键流程（端到端）

### 流程 A：系统触发

1. agent 调用 `msg_wait`。
2. 后端记录该 agent 进入 wait，并刷新活动/心跳。
3. 后台协调循环检查线程。
4. 发现“当前在线参与者全员 wait 且超时”。
5. 进入三分支处理：
	- 单在线当前管理员：发 `admin_takeover_confirmation_required`（human-only）。
	- 多在线全员 wait：发 `admin_coordination_timeout_notice`（human-only）并发 `admin_coordination_takeover_instruction` 给管理员。
	- 无可行动管理员：发 `agent_offline_risk_notice`（human-only）。

### 流程 B：人类决策

1. 人类在卡片点击按钮。
2. 前端调用 `/api/threads/{thread_id}/admin/decision`。
3. 后端校验来源卡片类型：
	- `admin_switch_confirmation_required` 只接受 `switch/keep`。
	- `admin_takeover_confirmation_required` 只接受 `takeover/cancel`。
4. 后端执行动作并写回决策状态。

---

## 6. 场景例子

### 例子 1：多在线，全员 wait

条件：

- M、N 在线，且都在 wait。
- 超过 `timeout_seconds`。

行为：

1. 发 `admin_coordination_timeout_notice`（human-only）。
2. 不发切换确认卡。
3. 系统发一条管理员协调指令（agent 可见），使管理员 `msg_wait` 返回并继续工作。

### 例子 2：单在线且就是当前管理员

条件：

- 只有 X 在线。
- X 是当前管理员。
- X 持续 wait 达到超时。

行为：

1. 发 `admin_takeover_confirmation_required`（human-only）。
2. 人类点 `takeover` 后，系统通知 X 立即接管并停止继续等待。
3. 人类点 `cancel` 后，不发送接管指令，继续观察其他 agent 是否恢复。

### 例子 3：当前管理员不可通知

条件：

- 触发超时，但当前管理员不在线或不在 wait，无法作为可行动接管者。

行为：

1. 系统仅发 `agent_offline_risk_notice`（human-only）。
2. 提示“当前 Thread 的 agent 可能全部掉线，请检查工作状态”。

---

## 7. 为什么这样设计

核心目标是“减少误判切换，同时保障线程可推进”。

1. 多在线时，先让管理员协调，避免频繁切换。
2. 单在线管理员时，让人类决定是否强制接管，避免盲目打断。
3. 无法通知管理员时，只做人类告警，避免发出无效 agent 指令。

---

## 8. 常见误解澄清

1. 误解：timeout 是“多久没人发消息”。
	- 事实：timeout 是“全体在线参与者处于 wait 的持续时长”。

2. 误解：只要超时就一定弹切管理员卡。
	- 事实：多在线场景默认不弹切换卡，改为通知管理员协调。

3. 误解：human-only 也能唤醒 agent。
	- 事实：不能。唤醒 `msg_wait` 只能靠 agent 可见消息。

---

## 9. 当前可操作结论

看到相关提示时建议按顺序处理：

1. 先看在线人数和触发类型（takeover 卡、timeout notice、offline risk）。
2. 若是单在线管理员卡：根据现场选择 `takeover` 或 `cancel`。
3. 若是多在线 timeout notice：检查管理员是否已收到协调指令并开始推进。
4. 若是 offline risk：优先检查 agent 进程在线状态与心跳。
